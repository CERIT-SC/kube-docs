# Jak spustit webový server NGINX

Pokoušet se spustit [NGINX](https://nginx.org/en/) webový server je běžný test pro nasazení aplikace/služby do Kubernetes. Zdá se, že je to snadný úkol, protože NGINX je jednoduché spustit a navíc existuje již připravený [docker image](https://hub.docker.com/_/nginx). Nicméně, nasazení může selhat z několika důvodů, proto poskytujeme několik poznámek, jak to provést na první pokus.

## Privilegovaný vs. Nepřevýšený

Tato Kubernetes platforma je multitenantní, tj. je sdílena mnoha uživateli a proto má určité omezení. Jedním z nich je nutnost nastavit [bezpečnostní kontext](../kubernetes/securitycontext), protože eskalace oprávnění není povolena. Nastavení bezpečnostního kontextu do nepřevýšeného režimu někdy vyžaduje vytvoření vlastního docker image pro aplikaci. Naštěstí pro NGINX již existuje připravený [neprivilegovaný image](https://hub.docker.com/r/nginxinc/nginx-unprivileged). Jeho referenční ID je: `nginxinc/nginx-unprivileged`. Tento image je plně kompatibilní s naší platformou tak, jak je, žádné změny v image nejsou potřeba.

## ID uživatele při spuštění

Výše zmíněný bezpečnostní kontext vyžaduje nastavení správného ID uživatele a ID skupiny (pole `RunAsUser`, `RunAsGroup`). Pro image `nginx-unprivileged` je správná hodnota pro obě `101`. Tato hodnota může být jednoduše získána pomocí `docker` následujícím způsobem. Viz `uid=101` a `gid=101`:

```shell
docker run -it --rm --entrypoint /bin/sh nginxinc/nginx-unprivileged
$ id
uid=101(nginx) gid=101(nginx) groups=101(nginx)
$
```

<Callout type="warning">
Tento příklad předpokládá, že image obsahuje `/bin/sh` a `id` binární soubory. Hlavně kontejnerizované aplikace v Go obvykle obsahují pouze Go aplikaci a nic jiného. V takovém případě je nutné konzultovat `Dockerfile` použitý pro sestavení image nebo dokumentaci projektu.
</Callout>

## Nasazení manifestu

Následující manifest může být použit pro spuštění instance NGINX. Název `nginx` může být nahrazen jakýmkoli jiným platným řetězcem (malá písmena, čísla, pomlčka). Také část s prostředky by měla být nastavena podle očekávaného použití (paměť a CPU).

```yaml filename="deployment.yaml" copy
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      securityContext:
        fsGroupChangePolicy: OnRootMismatch
        fsGroup: 101
        runAsNonRoot: true
        seccompProfile:
          type: RuntimeDefault
      containers:
      - name: nginx
        image: nginxinc/nginx-unprivileged
        imagePullPolicy: IfNotPresent
        securityContext:
          runAsUser: 101
          runAsGroup: 101
          allowPrivilegeEscalation: false
          capabilities:
            drop:
            - ALL
        resources:
          requests:
            cpu: 500m
            memory: 2Gi
          limits:
            cpu: 2
            memory: 8Gi
```

## Přidání trvalého úložiště

Výše uvedené nasazení bude fungovat, nicméně nebude poskytovat žádný užitečný obsah. Existuje pouze stránka `hello`. Také image neočekává, že jakýkoli obsah bude později přidán zkopírováním dat do webového kořenového adresáře (`/usr/share/nginx/html`), ten není ve výchozím nastavení zapisovatelný.

Optimální způsob, jak změnit poskytovaný obsah, je přidat PVC volume, více informací o [PVC](../kubernetes/pvc).

Můžeme vytvořit ad-hoc PVC pomocí následujícího manifestu:

```yaml filename="pvc.yaml" copy
kind: PersistentVolumeClaim
apiVersion: v1
metadata:
  name: test-nginx
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 2Gi
  storageClassName: nfs-csi
``` 

Data na tomto PVC jsou uchována, dokud není PVC smazáno.

Pokud bylo PVC vytvořeno, musíme změnit nasazení tak, aby bylo PVC použito:

```yaml {39-45}filename="deployment.yaml" copy
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      securityContext:
        fsGroupChangePolicy: OnRootMismatch
        fsGroup: 101
        runAsNonRoot: true
        seccompProfile:
          type: RuntimeDefault
      containers:
      - name: nginx
        image: nginxinc/nginx-unprivileged
        imagePullPolicy: IfNotPresent
        securityContext:
          runAsUser: 101
          runAsGroup: 101
          allowPrivilegeEscalation: false
          capabilities:
            drop:
            - ALL
        resources:
          requests:
            cpu: 500m
            memory: 2Gi
          limits:
            cpu: 2
            memory: 8Gi
        volumeMounts:
        - name: vol-1
          mountPath: /usr/share/nginx/html
      volumes:
      - name: vol-1
        persistentVolumeClaim:
          claimName: test-nginx
```

<Callout type="warning">
Název PVC (`name: test-nginx`) musí odpovídat `claimName` (`claimName: test-nginx`).
</Callout>

<Callout type="info">
Adresářový index není ve výchozím nastavení povolen pro NGINX. Proto, pokud není do PVC nahrán žádný soubor `index.html`, NGINX vrátí chybu `403 Forbidden`, pokud bude otevřena níže uvedená URL v prohlížeči.
</Callout>

## Exponování NGINX

Aby byl běžící NGINX přístupný z internetu, viz sekci [Exponování aplikací](../kubernetes/expose). V podstatě jsou potřeba dva další jednoduché manifesty:

### Služba

Manifest služby směruje provoz do běžícího Podu. Potřebuje správný cílový port a aplikační štítek. V našem případě jsme zvolili `app: nginx` jako aplikační štítek (viz sekce štítků metadat v manifestu nasazení), a z dokumentace `nginx-unprivileged` [dokumentace](https://hub.docker.com/r/nginxinc/nginx-unprivileged) vidíme, že NGINX používá port `8080`, takže manifest služby vypadá takto:

```yaml filename="service.yaml" copy
apiVersion: v1
kind: Service
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  ports:
    - port: 8080
      targetPort: 8080
      protocol: TCP
  selector:
    app: nginx
```

### Ingress

Poslední požadovaný manifest je manifest `Ingress`. Obsahuje hostname aplikace a může také poskytnout certifikát Let's Encrypt.

```yaml filename="ingress.yaml" copy
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: nginx
  annotations:
    kubernetes.io/tls-acme: "true"
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
spec:
  ingressClassName: nginx
  tls:
   - hosts:
       - "[some-name].dyn.cloud.e-infra.cz"
     secretName: [some-name]-dyn-cloud-e-infra-cz
  rules:
  - host: [some-name].dyn.cloud.e-infra.cz
    http:
      paths:
        - pathType: Prefix
          path: "/"
          backend:
            service:
              name: nginx
              port:
                number: 8080
```

Nahraďte `[some-name]` nějakým skutečným názvem. Nepoužívejte obecné názvy jako `test`, `example` a podobně, ty mohou být již obsazeny.

<Callout type="warning">
Název služby `name` a port `number` musí odpovídat hodnotám z objektu služby.
</Callout>
```
